electron 为什么要调用dll 呢？

electron 作用桌面端的开源的跨平台桌面应用程序开发框架，主要时依赖node Chromium 作为其渲染引擎和页面展示层 使用 Node.js 作为其后端运行环境，加入dll调用我们可以直接使用C++现成库，性能提升，特别时大数据和密集型计算场景，以及目前AI智能对话等方面。而且核心代码也能更好地保护起来。

方案一 N-API （C++）

N_API

Node-API | Node.js v21.7.1 Documentation​nodejs.org/api/n-api.html

Node-API | Node.js v21.7.1 Documentation Node-API | Node.js v21.7.1 Documentation Node-API | Node.js v21.7.1 Documentation


N-API 提供了一组稳定的 C/C++ API，使得插件开发者可以编写与 Node.js 的底层 API 无关的插件。这意味着插件可以在不同的 Node.js 版本之间共享，而无需重新编译或修改代码。此外，N-API 还提供了对操作系统抽象的功能，使得插件可以在不同的操作系统上运行，而不需要针对每个操作系统编写不同的代码


它是用来编写 C++ addons 大致流程如下：


  1. 编写 C++ 代码
  2. 使用 Node.js N-API
  3. 编写绑定代码：在 C++ 代码中，需要编写绑定代码来将 C++ 函数暴露给 JavaScript 代码调用。这通常使用 N-API 或者 NAN 提供的 API 来实现。
  4. 构建插件：编写完 C++ 代码和绑定代码后，需要将它们编译成共享库文件（例如 .dll、.so 或者 .dylib 文件），以供 Node.js 载入和使用。
  5. 在 Node.js 中使用插件：将编译好的插件文件（共享库文件）载入到 Node.js 中，并在 JavaScript 代码中调用插件提供的功能。

1.编写c++

#include <node_api.h>

napi_value Method(napi_env env, napi_callback_info info) {
  napi_status status;
  napi_value greeting;

  status = napi_create_string_utf8(env, "Hello from C++ addon!", NAPI_AUTO_LENGTH, &greeting);
  if (status != napi_ok) return nullptr;
  return greeting;
}

napi_value Init(napi_env env, napi_value exports) {
  napi_status status;
  napi_value fn;

  status = napi_create_function(env, nullptr, 0, Method, nullptr, &fn);
  if (status != napi_ok) return nullptr;

  status = napi_set_named_property(env, exports, "hello", fn);
  if (status != napi_ok) return nullptr;
  return exports;
}

NAPI_MODULE(NODE_GYP_MODULE_NAME, Init)

方案二 ffi-napi

这个方式很多人已经换成koffi 我会放到方案三种详细讲解，由于配置环境各方面、以及后续启动中遇到很多bug导致很多开发者放弃的原因的之一，其实所有问题你就记住一句话：事出必有因，解决它只要你肯花时间梳理并记录问题。


来自gpt的回答：ffi-napi 是一个 Node.js 模块，它提供了一个简单的方式来调用外部（本地）的 C/C++ 函数，而无需编写 C++ 插件。它使用了 Node.js 的 N-API 来实现跨版本的稳定性，并且具有良好的跨平台兼容性。

使用 ffi-napi，你可以在 Node.js 中直接调用已编译的动态链接库（.dll 文件或者 .so 文件），而无需编写任何 C++ 代码或者构建 C++ 插件。这对于需要调用现有的 C/C++ 代码库、或者对于不熟悉 C++ 编程的开发者来说是非常方便的

由于作者比较懒，很多调用其实是用python实现的，所有你在开发过程中需要用到python 把ffi-napi编译成二进制文件！这个时候就要用到node-gyp ，如果你是前端大佬并且熟悉node，肯定听说过 原生插件构建工具 node-gyp ————其实本身使用ffi-napi 不难，只要是使用 node-gyp 编译 ffi-napi 时环境的配置！

1. 编写C++

一个类似 javascript 的类调用

#include <iostream>
#include <string>

std::string token = "123456";

class cert {
public:
    cert() {}

    std::string readCert() {
        return token;
    }
};

extern "C" {
    cert* construct() {
        return new cert();
    }

    const char* readCert(cert* a) {
        return a->readCert().c_str();
    }

    void destruct(cert* a) {
        delete a;
    }
}


2.编译dll

g++ -shared -O3 -o cert.dll cert.cpp

3.开始调用

编写 cert.js 文件内容

const ffi = require('ffi-napi');
const path = require('path');

// 加载 DLL
const cert = ffi.Library(path.join(__dirname, 'cert'), {
    'construct': ['pointer', []],
    'readCert': ['string', ['pointer']],
    'destruct': ['void', ['pointer']]
});

// 创建 cert 实例
const certInstance = cert.construct();

// 调用 readCert 函数并输出结果
const token = cert.readCert(certInstance);
console.log('Token: ' + token);

// 销毁 cert 实例
cert.destruct(certInstance);


运行调用

node cert.js

为什么要这样做？

该方式引出了一个问题，就是编译好的dll 用ffi-api 或者其它方法无法调用报错

Dynamic Linking Error: Win32 Error 126

这个是一个比较常见而且对小白比较难解决的问题，网上各种解决方法都没有讲到这个问题。

一般出现这个报错说明以下几种情况：


3 和 4 一般不会出现，1，基本也不会，

其实大部分情况是：80% 是缺少dll 依赖 ，20%是路径错误，dll编译的位数不对，编译模式不对 debug 、release模式。

听起来是不是很复杂，其实你严格按照我的方式走不会出错的。

还是那句话 事出必有因
